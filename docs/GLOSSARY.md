# Glossary — Open Hoops

> Domain terms, system terms, and abbreviations used across this project.

---

## Basketball Domain Terms

| Term | Definition |
|---|---|
| **Court** | The playing surface, typically 28.65 m × 15.24 m (NBA) or 28 m × 15 m (FIBA). |
| **Half-court** | One half of the full court (one basket). Many drills and analytics focus on half-court. |
| **Full-court** | The entire playing surface from baseline to baseline. |
| **Paint / Key** | The rectangular area under the basket. Also called "the lane." |
| **Three-point arc** | The curved line beyond which shots count as 3 points. 7.24 m from the basket (NBA). |
| **Mid-range** | The area between the paint and the three-point arc. |
| **Corner three** | A three-point shot from the corner of the court (flatter arc, shorter distance). |
| **Transition** | A fast-break phase where the defense is not yet set. |
| **Possession** | A continuous sequence where one team controls the ball. |
| **Spacing** | How well players are distributed across the court to maximize scoring opportunities. |
| **Zone defense** | A defensive scheme where players guard areas rather than specific opponents. |
| **Man-to-man** | A defensive scheme where each player guards a specific opponent. |
| **Screen / Pick** | An offensive player setting a stationary block to free a teammate. |
| **Heatmap** | A spatial visualization showing density of player presence or activity on the court. |
| **Shot chart** | A visualization of shot locations and outcomes across a game or season. |

## Computer Vision Terms

| Term | Definition |
|---|---|
| **Detection** | Identifying the presence and bounding box of an object (player, ball) in a single frame. |
| **Tracking** | Associating detections across frames to maintain a persistent identity (track_id) per object. |
| **Bounding box (bbox)** | A rectangle [x1, y1, x2, y2] in pixel coordinates enclosing a detected object. |
| **Confidence score** | A 0–1 value indicating the model's certainty about a detection. |
| **Frame** | A single image from a video. Typically one of 30 per second at 30 fps. |
| **FPS** | Frames per second. Determines temporal resolution of detection. |
| **Homography** | A mathematical transform (3×3 matrix) that maps one 2D plane to another. Used here to map pixel coordinates to court coordinates. |
| **Keypoints** | Specific identifiable points (e.g., court corners, baseline intersections) used for calibration. |
| **YOLO** | "You Only Look Once" — a family of real-time object detection models. Used here via Ultralytics. |
| **ByteTrack** | A multi-object tracking algorithm that uses both high- and low-confidence detections to maintain tracks. |
| **BoT-SORT** | An alternative tracker combining ByteTrack with camera motion compensation. |
| **Occlusion** | When one object blocks another from the camera's view, causing detection to fail temporarily. |
| **Telemetry** | The structured output of the CV pipeline: per-frame detections with court coordinates and track IDs. |
| **Court coordinate** | A position on the court expressed in real-world meters (x, y) rather than pixels. |

## System Terms

| Term | Definition |
|---|---|
| **Job** | A single video processing request. Has a lifecycle: queued → processing → complete (or failed). |
| **Job queue** | Redis-backed task queue where pending CV jobs wait to be picked up by the CV Worker. |
| **CV Worker** | The Python service that performs frame extraction, detection, tracking, and homography. |
| **Analytics Worker** | The Python service that reads telemetry and computes player/team metrics. |
| **LLM Service** | The Python service that generates coaching reports using a local Ollama model. |
| **MinIO** | Local S3-compatible object storage for videos, telemetry files, and heatmap images. |
| **Ollama** | Local LLM server that runs quantized language models (Llama, Mistral, etc.) without cloud. |
| **API Gateway** | The FastAPI service that accepts requests from the frontend and coordinates backend services. |
| **Calibration** | The process of mapping known court keypoints in a video frame to real-world court coordinates. |
| **Homography matrix** | The 3×3 transform computed from calibration points. Stored per-job and applied during CV processing. |
| **Telemetry JSON** | The structured output file containing all detections + court coordinates for a processed video. |
| **Analytics summary** | Aggregated metrics computed from telemetry (distances, speeds, zone distributions, spacing). |
| **Coaching report** | Free-text analysis of analytics summary generated by the local LLM. |
| **Track ID** | A persistent integer identifier for a player/ball across multiple frames in the same video. |

## Agentic SDLC Terms

| Term | Definition |
|---|---|
| **Agentic SDLC** | A software development lifecycle where AI agents are the primary implementers, guided by Markdown specs. |
| **Phase** | A bounded unit of work with a defined scope, inputs, outputs, and completion criteria. |
| **Phase doc** | The Markdown file (`phases/PHASE_XX_*.md`) that specifies the tasks for a single phase. |
| **Source of truth** | In this project, Markdown files are always the source of truth for design decisions. |
| **Agent scope** | The bounded set of files, services, and tasks an agent is permitted to modify in a given phase. |
| **Owner review** | The human owner's confirmation that a phase is complete and correct before the next phase begins. |
